================================================================================
MODULE OVERVIEW
================================================================================

The Features Module transforms raw market data into meaningful trading signals:

INPUT:  LOBSnapshot, Tick, Trade objects from core/types.hpp
OUTPUT: Feature signals for strategy algorithms and ML models

5 Core Feature Components:
1. Microprice - Volume-weighted fair value
2. Order Flow Imbalance (OFI) - Momentum indicator
3. Spread Tracking - Bid-ask spread with rolling statistics
4. Trade Intensity - Activity level measurement
5. Online Normalizer - Feature standardization

All components:
- Zero dynamic allocations
- Cache-friendly design
- Incremental computation
- Numerically stable
- Sub-microsecond latency

================================================================================
QUICK REFERENCE
================================================================================

1. MICROPRICE
   Purpose: Volume-weighted fair value
   Formula: microprice = (bid_px * ask_vol + ask_px * bid_vol) / (bid_vol + ask_vol)
   Usage:   
   ┌─────────────────────────────────────────────┐
   │ Microprice mp;                              │
   │ double price = mp.update(lob_snapshot);     │
   │ bool valid = mp.is_valid();                 │
   └─────────────────────────────────────────────┘
   Performance: 10-20 ns, 16 bytes state

2. ORDER FLOW IMBALANCE (OFI)
   Purpose: Volume change momentum
   Formula: OFI = Δ(bid_volume) - Δ(ask_volume)
   Usage:
   ┌─────────────────────────────────────────────┐
   │ OrderFlowImbalance ofi;                     │
   │ int64_t ofi_val = ofi.update(lob);          │
   │ bool bullish = ofi.is_positive();           │
   │ int64_t imbal = ofi.imbalance();  // level  │
   └─────────────────────────────────────────────┘
   Performance: 5-10 ns, 32 bytes state

3. SPREAD TRACKING
   Purpose: Bid-ask spread with rolling statistics
   Features: Mean, StdDev, Min/Max, Z-score, Outlier detection
   Usage:
   ┌─────────────────────────────────────────────┐
   │ SpreadTracker<256> spreads;                 │
   │ spreads.update(lob);                        │
   │ double mean = spreads.mean();               │
   │ double z = spreads.z_score();               │
   │ bool wide = spreads.is_wide_outlier(2.0);   │
   │                                             │
   │ // Trend detection                          │
   │ SpreadWidthMonitor<256> monitor;            │
   │ monitor.update(lob);                        │
   │ if (monitor.is_widening()) { ... }          │
   └─────────────────────────────────────────────┘
   Performance: 50-100 ns, 2 KB for 256-tick window

4. TRADE INTENSITY
   Purpose: Trades per time window
   Formula: count = number of trades in (now - window_duration, now]
   Usage:
   ┌─────────────────────────────────────────────┐
   │ const uint64_t WINDOW_100MS = 100000000;    │
   │ TradeIntensity<1024> intensity(WINDOW_100MS)│
   │                                             │
   │ intensity.add_trade(trade_timestamp);       │
   │ uint32_t count = intensity.count();         │
   │ double rate = intensity.rate_per_second();  │
   │ int level = intensity.intensity_level();    │
   │                                             │
   │ // With moving average                      │
   │ TradeIntensityWithStats<1024,50> stats(...) │
   │ stats.update_window(current_time);          │
   │ bool high = stats.above_average();          │
   └─────────────────────────────────────────────┘
   Performance: 20-50 ns (add), O(1) query

5. ONLINE NORMALIZER (Welford's Algorithm)
   Purpose: Feature standardization with streaming stability
   Formula: z = (x - mean) / std_dev
   Usage:
   ┌─────────────────────────────────────────────┐
   │ OnlineNormalizer norm;                      │
   │ double z_score = norm.update(value);        │
   │ bool stable = norm.is_stable(100);          │
   │ bool anomaly = norm.is_anomalous(3.0);      │
   │                                             │
   │ // Multiple features                        │
   │ MultiFeatureNormalizer<10> multi;           │
   │ multi.update(values, normalized_values);    │
   │                                             │
   │ // Rolling window normalization             │
   │ RollingWindowNormalizer<256> rolling;       │
   │ rolling.update(value);                      │
   │ double z = rolling.normalize(test_value);   │
   └─────────────────────────────────────────────┘
   Performance: 20-30 ns, 40 bytes state

COMPLETE FEATURE PIPELINE
================================================================================

Orchestrate all features with FeaturePipeline class:

┌─────────────────────────────────────────────────────────────────────┐
│ FeaturePipeline<256, 256> pipeline;                                 │
│                                                                     │
│ // Per LOB update                                                   │
│ CompactFeatureSet features;                                         │
│ pipeline.update(lob_snapshot, features);                            │
│                                                                     │
│ // Per trade                                                        │
│ pipeline.add_trade(trade_timestamp);                                │
│                                                                     │
│ // Output: 64-byte cache-aligned feature set                        │
│ struct CompactFeatureSet {                                          │
│     double microprice;      // 8 bytes                              │
│     double spread;          // 8 bytes                              │
│     int64_t ofi;           // 8 bytes                              │
│     uint32_t trade_count;  // 4 bytes                              │
│     float norm_microprice; // 4 bytes (normalized)                  │
│     float norm_spread;     // 4 bytes (normalized)                  │
│     float norm_ofi;        // 4 bytes (normalized)                  │
│     int32_t padding;       // 4 bytes alignment                     │
│ };                                                                  │
└─────────────────────────────────────────────────────────────────────┘

Usage in Strategy:

void Strategy::on_lob_update(const LOBSnapshot& lob) {
    CompactFeatureSet features;
    pipeline.update(lob, features);
    
    // Raw signals
    assess_fair_value(features.microprice);
    assess_momentum(features.ofi);
    assess_liquidity(features.spread);
    
    // Normalized signals for ML
    ml_model.predict({
        features.norm_microprice,
        features.norm_ofi,
        features.norm_spread
    });
}

void on_price(double price) {
    double z = price_norm.update(price);
    
    if (std::abs(z) > ANOMALY_THRESHOLD) {
        log_anomaly("Price z-score: %.2f", z);
        pause_trading();
    }
}

================================================================================
DESIGN DECISIONS
================================================================================

1. WHY WELFORD'S ALGORITHM?
   Traditional: variance = E[X²] - E[X]² → UNSTABLE
   Welford's: incremental deltas → STABLE
   
   Example: For large numbers (~10⁸), traditional method loses precision
   Welford's maintains accuracy regardless of magnitude

2. WHY CIRCULAR BUFFERS?
   ✓ No allocations after initialization
   ✓ Deterministic memory usage
   ✓ Efficient replacement (single index increment)
   ✓ Works with fixed-size arrays
   
   Alternative: std::vector → allocations, exceptions, slower

3. WHY TIME-BASED PRUNING FOR TRADES?
   ✓ O(n) but n=count_of_old_trades, typically 0
   ✓ Trades stay in order
   ✓ Single memory layout
   
   Alternative: priority queue → O(log n) but more overhead

4. WHY 64-BYTE ALIGNED FEATURES?
   ✓ One cache line per structure
   ✓ No false sharing between cores
   ✓ Efficient memory layout
   ✓ CompactFeatureSet fits exactly one cache line

5. WHY INCREMENTAL COMPUTATION?
   ✓ O(1) per tick vs O(n) recalculation
   ✓ Suitable for streaming data
   ✓ Lower latency for ML models
   ✓ Enables real-time adaptation

INTEGRATION WITH CORE MODULE
================================================================================

Features Module uses Core Module components:

FROM core/types.hpp:
- LOBSnapshot: bid/ask prices and volumes
- Tick: market data updates
- Trade: executed trade details

FROM core/time.hpp:
- HighResolutionClock::now(): trade timestamps
- LatencyMeasurement: measure feature computation latency

FROM core/logger.hpp:
- HFT_LOG_INFO(): log feature events
- HFT_LOGF_WARN(): formatted logging for anomalies

FROM core/ring_buffer.hpp:
- SPSCQueue<CompactFeatureSet>: pass features between threads

Example Integration:

#include "core/types.hpp"
#include "core/time.hpp"
#include "features/features.hpp"

FeaturePipeline<256, 256> pipeline;

void market_data_thread() {
    while (true) {
        LOBSnapshot lob = receive_lob();
        
        CompactFeatureSet features;
        pipeline.update(lob, features);
        
        feature_queue.enqueue(features);
        
        HFT_LOGF_DEBUG("Spread: %.4f, OFI: %ld", 
                       features.spread, features.ofi);
    }
}

================================================================================
BEST PRACTICES
================================================================================

1. INITIALIZE EARLY
   - Create all feature computers at startup
   - Initialize normalizers before trading (need ~100 samples)

2. USE APPROPRIATE WINDOW SIZES
   - Powers of 2 for efficient computation
   - 256 ticks: ~0.25 seconds at 1000 Hz
   - Adjust for your trading frequency

3. MONITOR STABILITY
   - Check normalizer.is_stable(100) before ML usage
   - Track rolling window completeness
   - Allow warmup period in backtests

4. HANDLE EDGE CASES
   - Always check for NaN/infinity in features
   - Use try/catch around normalizer operations
   - Verify OFI doesn't overflow (use int64_t)

5. CACHE ALIGNMENT
   - CompactFeatureSet is 64-byte aligned
   - Pass by reference when possible
   - Minimize copies between threads

6. LATENCY MEASUREMENT
   - Use core/time.hpp LatencyMeasurement
   - Profile on target hardware
   - Account for JIT compiler warmup

7. THREAD SAFETY
   - Each feature computer: single-threaded
   - Use ring buffers for inter-thread communication
   - No shared state between threads
